100 Questions & Answers for QA Testers
Section 1: Basics of Crash Analytics
Q1. What is mobile app crash analytics?
A: Mobile app crash analytics is the process of collecting, recording, and analyzing crash data from
mobile applications to understand why they fail and how to fix them.
Q2. Why is crash analytics important for QA testers?
A: It helps QA testers identify issues before and after release, understand crash frequency, prioritize
bug fixes, and ensure app stability for end users.
Q3. What is a crash in a mobile app?
A: A crash is an unexpected termination of an application caused by an unhandled error, exception, or
system condition that the app cannot recover from.
Q4. What is the difference between a crash and a freeze?
A: A crash causes the app to close immediately, while a freeze (or ANR) causes the app to become
unresponsive without closing, requiring user action or system intervention.
Q5. What is crash-free user rate?
A: Crash-free user rate is the percentage of users who did not experience a crash during a given time
period. A rate above 99% is generally considered acceptable.
Q6. What is crash rate?
A: Crash rate is the number of crashes divided by the total number of app sessions, usually expressed
as a percentage. It measures how often the app crashes.
Q7. What is a stack trace?
A: A stack trace is a report of the active method calls at the time a crash occurred. It shows the
sequence of function calls that led to the error.
Q8. What does ANR stand for?
A: ANR stands for Application Not Responding. It occurs on Android when the main thread is blocked
for more than 5 seconds.
Q9. What is a fatal crash vs a non-fatal crash?
A: A fatal crash causes the app to terminate immediately. A non-fatal crash (or exception) is caught and
logged but does not stop the app.
Q10. What is a crash report?
A: A crash report is an automatically generated document that contains details about a crash, including
the stack trace, device info, OS version, and user actions before the crash.
Section 2: Reading Stack Traces
Q11. How do you read a stack trace?
A: Read a stack trace from top to bottom. The first line shows the exception type and message. The
lines below show the call hierarchy, starting from where the crash occurred down to the original method
call.
Q12. What is a NullPointerException?
A: A NullPointerException (NPE) occurs when the app tries to use a reference variable that points to
null (no object). It is one of the most common Android crashes.
Q13. What is an OutOfMemoryError?
A: An OutOfMemoryError occurs when the app requests more memory than the device can provide,
often caused by memory leaks or loading large images/data without optimization.
Q14. What is an IndexOutOfBoundsException?
A: It occurs when code tries to access an array or list element using an index that is outside the valid
range (e.g., accessing index 5 in a list of 3 items).
Q15. What does 'Caused by' mean in a stack trace?
A: 'Caused by' indicates the root exception that triggered the crash. It is often more useful than the
outer exception for diagnosing the actual problem.
Q16. What is a ClassCastException?
A: A ClassCastException occurs when the app tries to cast an object to a type it is not compatible with,
resulting in a runtime error.
Q17. What is an IllegalStateException?
A: An IllegalStateException occurs when a method is called at an inappropriate time or when the app is
in an invalid state for that operation.
Q18. What is a StackOverflowError?
A: A StackOverflowError happens when there is infinite or excessive recursion, causing the call stack to
exceed its maximum size.
Q19. What does the line number in a stack trace tell you?
A: The line number tells you exactly where in the source code the crash occurred, helping developers
locate the bug quickly.
Q20. What is a thread dump?
A: A thread dump is a snapshot of all currently running threads in an application at a specific point in
time, often used to diagnose ANR issues.
Section 3: Android Crash Analysis
Q21. What is Logcat?
A: Logcat is Android's logging system that captures real-time log messages from the device, including
crash logs, system warnings, and debug info.
Q22. How do you filter crash logs in Logcat?
A: You can filter Logcat by log level (Error), by tag (e.g., AndroidRuntime), or by package name to see
only relevant crash messages.
Q23. What is ADB and how is it used for crash analysis?
A: ADB (Android Debug Bridge) is a command-line tool that lets QA testers connect to Android devices,
collect logs, and run debugging commands.
Q24. What command shows crash logs in ADB?
A: The command 'adb logcat *:E' filters logs to show only error-level messages, which typically includes
crash information.
Q25. What is a tombstone file in Android?
A: A tombstone file is a detailed crash report generated by the Android OS when a native code crash
(C/C++) occurs. It contains memory addresses, registers, and stack traces.
Q26. What is a native crash in Android?
A: A native crash occurs in C/C++ code (NDK layer) and produces a signal like SIGSEGV or SIGABRT.
These are harder to debug than Java crashes.
Q27. What is SIGSEGV?
A: SIGSEGV (Segmentation Fault) is a signal sent when a program tries to access a memory location it
is not allowed to access, causing a native crash.
Q28. How do you reproduce an Android crash from a report?
A: Match the device model, OS version, and app version from the report. Follow the steps listed before
the crash, and replicate the exact user flow that triggered it.
Q29. What is a ProGuard mapping file?
A: A ProGuard mapping file translates obfuscated class and method names back to their original
names, making crash stack traces readable after code minification.
Q30. What is a deobfuscated stack trace?
A: A deobfuscated stack trace is a crash report where obfuscated code names have been converted
back to the original readable source code names using a mapping file.
Section 4: iOS Crash Analysis
Q31. How are iOS crashes reported?
A: iOS crashes are reported through Xcode's Organizer, Apple's built-in crash reporter, or third-party
tools like Firebase Crashlytics or Sentry.
Q32. What is a .ips file in iOS?
A: A .ips file (Incident Process State) is Apple's crash report format that contains the exception type,
stack trace, and device info for an iOS crash.
Q33. What is a dSYM file?
A: A dSYM (Debug Symbol) file maps memory addresses in crash reports back to readable source
code lines. It is required to symbolicate iOS crash reports.
Q34. What is symbolication in iOS?
A: Symbolication is the process of converting memory addresses in a crash report into readable
function names and line numbers using the dSYM file.
Q35. What is EXC_BAD_ACCESS in iOS?
A: EXC_BAD_ACCESS is an iOS crash signal that occurs when the app tries to access memory that
has already been freed or is otherwise invalid.
Q36. What is a watchdog termination in iOS?
A: A watchdog termination occurs when iOS kills an app that has been unresponsive for too long
(usually during launch or background transitions).
Q37. What does SIGABRT mean in iOS crashes?
A: SIGABRT (Signal Abort) means the app deliberately terminated itself, often due to an assertion
failure, an uncaught exception, or calling abort() directly.
Q38. What is Xcode Organizer used for in crash analysis?
A: Xcode Organizer is a tool in Xcode that shows crash reports and energy logs from devices that have
opted in to share diagnostics with developers.
Q39. What is TestFlight and how does it relate to crash analysis?
A: TestFlight is Apple's beta testing platform. Crashes during TestFlight testing are logged and can be
reviewed by developers through App Store Connect.
Q40. What is MetricKit in iOS?
A: MetricKit is an Apple framework that delivers on-device app diagnostics including crash logs, hang
reports, and performance metrics directly to the app.
Section 5: Crash Analytics Tools
Q41. What is Firebase Crashlytics?
A: Firebase Crashlytics is a lightweight, real-time crash reporting tool from Google that helps track,
prioritize, and fix app crashes for Android and iOS.
Q42. How does Crashlytics group crashes?
A: Crashlytics groups crashes by their stack trace signature, creating 'issues' that cluster similar
crashes together for easier analysis and prioritization.
Q43. What is Sentry?
A: Sentry is an open-source error and crash monitoring platform that provides real-time crash reports,
breadcrumbs (user actions before crash), and performance monitoring.
Q44. What are breadcrumbs in crash analytics tools?
A: Breadcrumbs are a trail of events recorded before a crash occurs, such as user actions, navigation
changes, and network requests, helping reproduce the crash path.
Q45. What is Bugsnag?
A: Bugsnag is a crash monitoring tool that detects errors in mobile, web, and server apps and provides
detailed crash diagnostics including user journey and device data.
Q46. What is AppCenter (Microsoft)?
A: App Center is a Microsoft platform that offers crash analytics, distribution, and automated testing for
Android and iOS apps.
Q47. What is Instabug?
A: Instabug is a mobile testing platform that provides in-app bug reporting, crash reporting, and user
feedback features for mobile QA teams.
Q48. How do you integrate Firebase Crashlytics into an Android app?
A: Add the Crashlytics Gradle plugin and dependency to your build files, apply the plugin in your
app-level build.gradle, and sync the project. No additional code is needed for automatic crash reporting.
Q49. What is the difference between Crashlytics and Sentry?
A: Crashlytics is simpler and tightly integrated with Firebase/Google. Sentry supports more platforms,
provides richer context (breadcrumbs, releases), and offers both cloud and self-hosted options.
Q50. How do you test if crash reporting is working?
A: Use a test crash method (e.g., Crashlytics.getInstance().crash() for Android) to force a crash and
verify it appears in the dashboard within a few minutes.
Section 6: Crash Prioritization
Q51. How do you prioritize crashes?
A: Prioritize crashes based on crash rate, number of affected users, severity of the user impact, crash
frequency, and whether it affects critical app flows.
Q52. What is crash severity?
A: Crash severity measures how badly a crash impacts the user experience. A crash on the checkout
page is more severe than one on a settings screen.
Q53. What is crash volume?
A: Crash volume is the total number of crash occurrences within a given time period. High volume
crashes should be prioritized even if individual severity seems low.
Q54. What is a P1 crash?
A: A P1 (Priority 1) crash is a critical crash that affects core functionality (e.g., app launch, login,
payment), impacting a large number of users and requiring immediate attention.
Q55. What is the impact score in crash analytics?
A: An impact score combines crash frequency and the number of affected users to give a single number
indicating how critical a crash is to address.
Q56. Should you always fix the most frequent crash first?
A: Not always. A less frequent crash on a critical user path (like payment or login) may need to be fixed
before a more frequent but minor crash on a rarely used screen.
Q57. What is a crash trend?
A: A crash trend shows how crash volume changes over time, helping identify if crashes are increasing
(regression) or decreasing (fix is working) after a release.
Q58. How do OS version and device type affect prioritization?
A: If a crash only affects a small percentage of devices or an old OS version, it may be lower priority. If
it affects the most popular device or latest OS, it is higher priority.
Q59. What is a regression crash?
A: A regression crash is a new crash introduced in a recent app update that did not exist in the previous
version. These are typically high priority.
Q60. How do you identify regression crashes?
A: Compare crash reports between the current and previous app versions. A crash that appears only in
the new version and not the old one is likely a regression.
Section 7: Bug Reporting
Q61. What should a good crash bug report include?
A: A good crash report should include: crash title, steps to reproduce, expected vs actual behavior,
device info, OS version, app version, stack trace, screenshots or screen recording, and frequency.
Q62. What is a steps-to-reproduce section?
A: Steps to reproduce is a numbered list of actions a developer can follow to trigger the crash. Clear
steps help developers confirm and fix the issue faster.
Q63. What is expected vs actual behavior in a bug report?
A: Expected behavior describes what should happen, and actual behavior describes what happens
instead (the crash). This helps developers understand the impact.
Q64. Why is device and OS info important in a crash report?
A: Some crashes are device or OS-specific. Providing the exact device model and OS version helps
developers reproduce the crash in the right environment.
Q65. How should you report a crash that is hard to reproduce?
A: Include the stack trace from the analytics tool, the session context, breadcrumbs, and any patterns
you've noticed (e.g., only happens with poor network). Mark it as intermittent.
Q66. What is a crash frequency in a bug report?
A: Crash frequency indicates how often the crash occurs: always, intermittently, or rarely. This helps
developers and managers understand reproducibility and urgency.
Q67. Should you attach a screen recording to a crash bug report?
A: Yes, whenever possible. A screen recording showing the exact user actions before the crash is
extremely helpful for reproduction and validation.
Q68. What is a bug title best practice?
A: A good bug title follows the format: '[Feature] Action causes crash under condition', e.g., '[Checkout]
App crashes when applying promo code with empty cart'.
Q69. What labels or tags should you add to a crash bug?
A: Use tags like the affected platform (Android/iOS), severity (P1/P2), type (crash), affected feature,
app version, and OS version for easy filtering and tracking.
Q70. When should a crash be marked as resolved?
A: A crash should be marked resolved after the developer submits a fix, QA verifies the fix in a test
build, and the crash rate drops to zero in production after the fix release.
Section 8: Root Cause Analysis
Q71. What is root cause analysis (RCA) for crashes?
A: RCA is the process of investigating a crash to find its underlying cause rather than just treating the
symptoms, preventing the crash from recurring.
Q72. How do you find the root cause of a NullPointerException?
A: Identify which variable is null in the stack trace, trace back to where it is assigned or initialized, and
determine why it was not assigned a valid value.
Q73. What role do logs play in root cause analysis?
A: Logs provide context about what happened before the crash. Verbose logs, warnings, and error
messages can reveal the state that triggered the problem.
Q74. What is a memory leak and how does it cause crashes?
A: A memory leak occurs when an app holds references to objects that are no longer needed,
preventing garbage collection and eventually causing an OutOfMemoryError crash.
Q75. How can threading issues cause crashes?
A: Accessing UI elements from a background thread, race conditions, and improper synchronization
can all cause crashes. Android requires UI updates to happen on the main thread.
Q76. What is a race condition?
A: A race condition occurs when two threads access or modify shared data at the same time, leading to
unpredictable behavior and potential crashes.
Q77. How do configuration changes cause crashes in Android?
A: Screen rotations and language changes trigger activity recreation. If data is not properly saved and
restored, it can cause crashes due to null references.
Q78. What are common causes of iOS crashes?
A: Common iOS crash causes include force unwrapping nil optionals, memory management errors
(retain cycles), accessing deallocated objects, and main thread violations.
Q79. What is a force unwrap crash in Swift?
A: In Swift, using the ! operator to force unwrap an optional that contains nil causes an immediate crash
with a 'Fatal error: unexpectedly found nil' message.
Q80. How do you use version comparison to find the root cause?
A: Compare the code changes between the last stable version and the crashing version using a diff or
changelog to identify what new code may have introduced the crash.
Section 9: Crash Prevention & Best Practices
Q81. What is defensive coding in relation to crash prevention?
A: Defensive coding means writing code that anticipates and handles potential errors gracefully, using
null checks, try-catch blocks, and input validation to prevent crashes.
Q82. How does unit testing help prevent crashes?
A: Unit tests verify that individual code units behave correctly under various conditions, catching
potential crash scenarios before the code reaches QA or production.
Q83. What is crash monitoring in production?
A: Crash monitoring in production involves using tools like Crashlytics or Sentry to automatically detect,
log, and alert teams about crashes experienced by real users.
Q84. What is proactive crash testing?
A: Proactive crash testing means deliberately testing edge cases, invalid inputs, poor network
conditions, and low memory scenarios to find crashes before users do.
Q85. How does code review help reduce crashes?
A: Code review allows peers to spot risky patterns like null dereferences, improper thread usage, or
missing error handling before the code is merged and tested.
Q86. What is monkey testing?
A: Monkey testing is automated random input testing that simulates unpredictable user behavior to find
crashes and ANRs that might not be covered by scripted test cases.
Q87. What is a crash alert threshold?
A: A crash alert threshold is a configured limit in crash monitoring tools that triggers a notification when
crash rate or volume exceeds a set value, prompting immediate investigation.
Q88. What is a rollback and when should you trigger one?
A: A rollback reverts the app to a previous version. It should be triggered when a new release
introduces a critical crash spike that cannot be fixed quickly with a hotfix.
Q89. How do you validate a crash fix?
A: Test the exact steps that caused the crash on the same device and OS version. Confirm the crash
no longer occurs. Monitor the crash dashboard after the fix goes to production.
Q90. What is a hotfix release?
A: A hotfix is an emergency app update released quickly to address a critical crash or bug affecting
production users, bypassing the normal release cycle.
Section 10: Advanced & Miscellaneous
Q91. What is a handled exception vs an unhandled exception?
A: A handled exception is caught using try-catch and the app continues running. An unhandled
exception is not caught, causing the app to crash immediately.
Q92. What is out-of-memory (OOM) analysis?
A: OOM analysis investigates why the app ran out of memory, typically by examining heap dumps,
memory profiler data, and identifying large objects or leaks.
Q93. What is a heap dump?
A: A heap dump is a snapshot of the app's memory at a specific moment, showing all objects and their
sizes. It is used to identify memory leaks causing OOM crashes.
Q94. What is the difference between a crash and an exception?
A: All crashes are caused by exceptions, but not all exceptions cause crashes. Exceptions that are
caught and handled do not crash the app.
Q95. What is a cold start crash?
A: A cold start crash occurs when the app crashes immediately on launch, often before the main screen
appears. These are very high severity as they prevent all users from using the app.
Q96. What is a background crash?
A: A background crash occurs when the app crashes while it is not in the foreground, often during
background tasks like syncing data or receiving push notifications.
Q97. What is a session in crash analytics?
A: A session is a single continuous period of app usage by a user, typically from when the app opens to
when it closes or goes to background for more than 30 minutes.
Q98. What is a crash-free session rate?
A: Crash-free session rate is the percentage of app sessions that completed without a crash. It is
different from crash-free user rate, which counts unique users.
Q99. How do network errors relate to crash analytics?
A: Network errors can trigger crashes if the app does not handle failure responses, timeouts, or null
data gracefully. Network state is often logged as context in crash reports.
Q100. What is the role of a QA tester in crash analytics post-release?
A: After release, QA testers monitor crash dashboards, triage new crashes, verify that known crashes
remain fixed, and investigate spikes in crash rate to report to the development team.
